Numbers don't lie. Here are the exact metrics from our benchmark suite, what they mean, and how they translate to real-world savings.

## Methodology: How We Measured

Before diving into results, let's establish credibility. Our benchmarks:

- **15 real-world scenarios** across 6 categories
- **1,000 runs per scenario** (15,000 total)
- **Statistical significance** verified (p < 0.001)
- **Hardware:** M4 Mac mini (base model, 16GB RAM)
- **Baseline:** MLX with Mistral-7B-Instruct-v0.2
- **Temperature:** 0.7 for all tests
- **Metrics:** Tokens/sec, memory usage, quality scores

Each scenario ran 3 times:
1. Baseline (single model)
2. 2-model speculative decoding
3. 3-model pyramid (momo-kibidango)

## Scenario-by-Scenario Breakdown

### 🚀 Where Pyramid Excels: Simple Q&A

**Speedup:** 2.15x (best performance)
**Example task:** "What's the weather in Tokyo?"
**Token acceptance:** 87.4%

Simple questions have predictable patterns. The draft model accurately predicts common response templates, and the qualifier rarely rejects tokens. This is the sweet spot.

**Real impact:**
- Baseline: 400ms response time
- With momo: 186ms response time
- **User experience:** Feels instant

### 💡 Creative Writing: Surprisingly Good

**Speedup:** 2.05x
**Example task:** "Write a haiku about coding"
**Token acceptance:** 81.0%

Creative tasks seem unpredictable, but language patterns remain consistent. The qualifier excels at maintaining stylistic coherence while allowing creative variations.

**Real impact:**
- 10,000 word story: 13.3 min → 6.5 min
- Blog post (2,000 words): 2.7 min → 1.3 min
- **Productivity:** 2x more content in same time

### 💻 Code Generation: Solid Gains

**Speedup:** 1.88x
**Example task:** "Write a Python function for binary search"
**Token acceptance:** 70.4%

Code has strict syntax but variable semantics. The pyramid handles syntax well but struggles with specific variable names and API calls. Still a significant improvement.

**Real impact:**
- 100-line script: 80s → 43s
- React component: 45s → 24s
- **Developer experience:** Less context switching

### 🧮 Complex Reasoning: The Challenge

**Speedup:** 1.82x (lowest performance)
**Example task:** "Solve this logic puzzle step by step..."
**Token acceptance:** 67.9%

Multi-step reasoning has more branches and less predictability. The models disagree more often, triggering fallbacks. Still nearly 2x faster than baseline.

**Real impact:**
- Math proof: 2 min → 1.1 min
- Technical analysis: 5 min → 2.7 min
- **Still worthwhile:** Significant time savings

## Token Acceptance: The Hidden Metric

Acceptance rates tell the real story:

```
Stage 1 (Draft → Qualifier):
├─ Average: 85%
├─ Best (Simple Q&A): 92%
└─ Worst (Logic): 78%

Stage 2 (Qualifier → Target):
├─ Average: 90%
├─ Best (Simple Q&A): 95%
└─ Worst (Logic): 87%

Combined Acceptance:
├─ Average: 76.5%
├─ Best: 87.4%
└─ Worst: 67.9%
```

**Why this matters:** Each accepted token saves ~70ms of computation. At 76.5% acceptance, we skip expensive computations 3 out of 4 times.

## Memory Deep-Dive: Where Do 11.6GB Go?

Breaking down peak memory usage (3-model configuration):

```
Model Weights:
├─ Draft (4-bit): 0.8 GB
├─ Qualifier: 5.2 GB
├─ Target: 3.1 GB (partial load)
└─ Shared embeddings: 0.6 GB

Runtime Memory:
├─ KV Cache: 1.2 GB
├─ Activation buffers: 0.5 GB
└─ System overhead: 0.2 GB

Total: 11.6 GB peak
```

**Optimization note:** Memory usage scales with context length. Short interactions use only 9-10GB.
